Tetris Hybrid — ベアメタル Embedded Swift 徹底解説

Raspberry Pi Pico (RP2040) 上で C言語ゼロ・100% Swift で実装されたテトリスゲーム。
SysTick 割り込みによるBGM再生、GPIO 割り込みによるボタンキャプチャ、協調ループによるゲームロジック・描画というハイブリッドアーキテクチャを採用。

1. アーキテクチャ概要

1.1 起動シーケンス

電源ON │ ▼ Boot ROM (RP2040内蔵) │ XIPフラッシュの先頭256バイトを読む ▼ Boot2 (Boot2.swift → .boot2セクション) │ XIPフラッシュコントローラを初期化 ▼ ベクターテーブル (.vectorセクション) │ エントリ0: スタックポインタ = 0x2004_0000 │ エントリ1: Reset_Handler ▼ Reset_Handler (VectorTable.swift) │ initializeMemorySections() → .data をROMからRAMにコピー、.bss をゼロクリア │ Application.main() ▼ Board.init() │ Clocks: XOSC → PLL → 125MHz │ SysTick: 1ms割り込み開始 │ ペリフェラルリセット + 初期化 ▼ メインループ開始

1.2 ハイブリッド設計

このプロジェクトの核心は、時間的に重要な処理を割り込みで、それ以外を協調ループで処理するハイブリッドアーキテクチャです。

┌─────────────────────────────────────────────┐ │ SysTick 割り込み (1ms) │ │ ┌─────────────┐ ┌──────────────────────┐ │ │ │ _tick += 1 │ │ Music.advanceIn...() │ │ │ │ (時刻カウンタ) │ │ (BGMノート切り替え) │ │ │ └─────────────┘ └──────────────────────┘ │ ├─────────────────────────────────────────────┤ │ GPIO 割り込み (IRQ13) │ │ ┌──────────────────────────────────────┐ │ │ │ ButtonISR.handle() │ │ │ │ (ボタン押下をビットマスクに記録) │ │ │ └──────────────────────────────────────┘ │ ├─────────────────────────────────────────────┤ │ 協調ループ (メインスレッド) │ │ ┌──────────┐ ┌──────────┐ ┌───────────┐ │ │ │ゲームロジック│ │ 描画処理 │ │ I2Cフラッシュ│ │ │ │ (高速) │ │ (高速) │ │ (~25ms) │ │ │ └──────────┘ └──────────┘ └───────────┘ │ └─────────────────────────────────────────────┘

なぜハイブリッドが必要か：

SSD1306 OLEDへのI2Cフラッシュに約25msかかります。この間メインスレッドはブロックされますが、割り込みは発火し続けます。BGM（1ms精度でノート切り替え）とボタン入力（押した瞬間を逃さない）を割り込みに置くことで、I2Cブロック中も正確に処理されます。

ボタンキャプチャGPIO割り込み即時I2C中の押下を逃さない
ゲームロジック協調ループ~16ms処理量が多い

3. ハードウェア層

3.1 MMIO レジスタアクセス

全てのハードウェアアクセスの基盤となる Register 型です。Swift の VolatileMappedRegister をラップし、コンパイラによる最適化（読み書きの省略・並べ替え）を防ぎます。

struct Register {
  let address: UInt32

  func load() -> UInt32 {
    VolatileMappedRegister<UInt32>(unsafeBitPattern: UInt(address)).load()
  }
  func store(_ value: UInt32) {
    VolatileMappedRegister<UInt32>(unsafeBitPattern: UInt(address)).store(value)
  }

  // RP2040 アトミックビット操作エイリアス
  func set(_ bits: UInt32)   { /* base + 0x2000 */ }  // ビットをセット
  func clear(_ bits: UInt32) { /* base + 0x3000 */ }  // ビットをクリア
  func xor(_ bits: UInt32)   { /* base + 0x1000 */ }  // ビットをトグル
}
RP2040のアトミックエイリアス：通常のRead-Modify-Writeでは割り込みとの競合が起きますが、RP2040はアドレスに+0x2000/+0x3000/+0x1000を加えるだけでアトミックなビット操作が可能です。割り込み無効化が不要です。

3.2 クロック・PLL・XOSC

RP2040を12MHz水晶から125MHzに引き上げるクロックツリーです。

XOSC (12MHz水晶) │ ▼ PLL_SYS (×125, ÷6, ÷2 = 125MHz) │ ├──→ clk_sys = 125MHz (CPU, バス) ├──→ clk_ref = 12MHz (ウォッチドッグ等) └──→ clk_peri = 125MHz (UART, I2C)

3.3 SysTick タイマー

ARM Cortex-M0+ 内蔵の24ビットダウンカウンタ。1msごとに割り込みを発生させます。

enum SysTick {
  nonisolated(unsafe) static var _tick: UInt32 = 0

  // 揮発性読み取り（ポインタ経由でコンパイラキャッシュを防止）
  static var milliseconds: UInt32 {
    withUnsafePointer(to: &_tick) { ptr in
      UnsafeRawPointer(ptr).loadUnaligned(as: UInt32.self)
    }
  }

  static func initialize() {
    rvr.store(125_000 - 1)  // 125MHz ÷ 125,000 = 1kHz (1ms)
    csr.store(0x7)           // ENABLE + TICKINT + CLKSOURCE
  }
}
遭遇した問題：揮発性変数のキャッシュ
_tick を直接読むと、コンパイラが値をレジスタにキャッシュし、割り込みによる更新が見えなくなります。withUnsafePointer + loadUnaligned で毎回メモリから読むことを強制しています。Swiftには C の volatile に相当する機能がないため、この回避策が必要です。

3.5 I2C コントローラ

DesignWare I2C IPブロック。400kHzマスターモードでSSD1306 OLEDと通信します。

IC_TAR の書き込み制約：ターゲットアドレスレジスタ (IC_TAR) はコントローラが無効 (IC_ENABLE=0) の間しか書き込めません。有効なまま書くと間欠的にI2C通信が失敗します。 
private static func setTarget(_ address: UInt8) {
  if address == currentTarget { return }  // キャッシュで冗長サイクルを回避

  enable.store(0)                          // 無効化
  while enableStatus.load() & 1 != 0 {}   // 無効化完了を待つ

  targetAddress.store(UInt32(address))     // IC_TAR 書き込み
  currentTarget = address

  enable.store(1)                          // 再有効化
}

3.6 SSD1306 OLED ディスプレイ

128×64ピクセルの単色OLED。物理的には横長ですが、ポートレートモード（64×128）で使用しています。

// 座標変換: 論理(x,y) → 物理(y, 63-x)
if SSD1306.portrait {
  px = y
  py = 63 - x
}

フレームバッファ（1024バイト）をRAMに保持し、flush() でI2C経由で全画面転送します。この転送に約25msかかることが、ハイブリッドアーキテクチャが必要な根本的理由です。

3.7 PWM（ブザー）

GP20に接続されたブザーでBGMを再生します。PWMスライス2、チャンネルA。

// クロック分周: 125MHz ÷ 16 = 7.8125MHz
// 周波数制御: freq = 7,812,500 ÷ top
// 例: 659Hz (E5) → top = 7,812,500 ÷ 659 = 11,855
PWM.setTop(slice: 2, top: 11855)
PWM.setDuty(pin: 20, duty: 11855 / 2)  // 50%デューティ = 最大音量

3.8 UART デバッグ出力

GP0(TX)/GP1(RX)で115200bps。Debug Probeのシリアルポート経由でPCに出力。

3.9 NVIC 割り込みコントローラ

ARM Cortex-M0+ の Nested Vectored Interrupt Controller。IO_BANK0 (IRQ13) を有効化してGPIOボタン割り込みを受け取ります。

4. ボード抽象化層

Board.swift がハードウェア初期化を一元管理します。

struct Board {
  var display = Display()

  init() {
    Clocks.initialize()            // 125MHz クロック設定
    SysTick.initialize()           // 1ms 割り込みタイマー

    // ペリフェラルリセット（デバッグ再デプロイの安定性）
    Resets.reset(.i2c0)
    Resets.reset(.uart0)
    Resets.reset(.pwm)
    Resets.unreset(.ioBank0)
    Resets.unreset(.padsBank0)

    UART.initialize(tx: 0, rx: 1)

    // I2Cパッド設定（直接書き込み、R-M-Wではない）
    PadsBank.store(16, 0x4A)       // SDA
    PadsBank.store(17, 0x4A)       // SCL
    IOBank.setFunction(16, .i2c)
    IOBank.setFunction(17, .i2c)

    IOBank.setFunction(20, .pwm)   // ブザー
    initButtons()                   // GP2-GP7
    ButtonISR.initialize()         // GPIO割り込み設定

    SSD1306.portrait = true
    display.initialize()
  }
}

5. ゲームロジック

5.1 Tetris — 盤面管理

10列×20行のテトリス盤面。各行を UInt16 のビットマスクで表現します（下位10ビット使用）。

// 行データアクセス（タプルをポインタでインデックスアクセス）
nonisolated(unsafe) static var boardRows:
  (UInt16, UInt16, ..., UInt16)  // 20要素のタプル
  = (0, 0, ..., 0)

static func getRow(_ r: Int) -> UInt16 {
  withUnsafePointer(to: &boardRows) { ptr in
    UnsafeRawPointer(ptr).assumingMemoryBound(to: UInt16.self)[r]
  }
}
なぜ配列ではなくタプルか：Embedded Swift で [UInt16](repeating: 0, count: 20) はヒープ確保が必要です。タプルならスタック/静的メモリに配置され、ヒープアロケーションを回避できます。

5.2 Piece — テトロミノ定義

7種のテトロミノ（I, O, T, S, Z, L, J）×4回転 = 28パターン。各パターンは4×4グリッドを UInt16 ビットマスクで表現します。

// ビットレイアウト: bit n = row(n/4), col(n%4)
// T-piece, rotation 0:
// . # . .    →  0b 0000_0000_0111_0010  →  0x0072
// # # . .
// . . . .
// . . . .

static func isFilled(type: UInt8, rotation: UInt8, row: Int, col: Int) -> Bool {
  let s = shape(type: type, rotation: rotation)
  return s & (1 << (row * 4 + col)) != 0
}

5.3 Music — BGM再生

コロブチカ（テトリスのテーマ曲）をPWMブザーで再生。SysTick割り込みから1msごとに呼ばれるため、I2Cフラッシュ中もBGMが途切れません。

// ノートデータ: 上位16ビット=周波数(Hz), 下位16ビット=長さ(ms)
// 例: (659 << 16) | 420  →  E5音, 四分音符420ms
static func noteData(_ i: Int) -> UInt32 {
  switch i {
  case 0: return (659 << 16) | 420   // E5 四分音符
  case 1: return (494 << 16) | 210   // B4 八分音符
  ...
  }
}

テンポ設定（ゲームボーイ版準拠）

音価長さBPM換算
八分音符210ms≈ BPM 143
四分音符420ms
付点四分音符630ms

ノート間に20msのギャップ（アーティキュレーション）を入れてスタッカート感を出しています。

5.4 ButtonISR — 割り込みボタン入力

GPIO落下エッジ割り込みでボタン押下を即座にキャプチャします。

// 割り込みハンドラ（IO_BANK0 IRQ13から呼ばれる）
static func handle() {
  if IOBank.isInterruptPending(pinLeft, .fallingEdge) {
    IOBank.acknowledgeInterrupt(pinLeft, .fallingEdge)
    pressed |= 1  // ビットマスクに記録
  }
  // ... 他のボタンも同様
}

// メインループから呼ばれる（読み取り＋クリア）
static func consumePressed() -> UInt8 {
  let val = pressed
  pressed = 0
  return val
}
ポーリングとISRの併用：メインループではポーリング（Buttons.left）による保持検出とISR（ButtonISR.consumePressed()）による押下検出を組み合わせています。ただし、回転やハードドロップはISRのバウンスノイズで二重発動するため、ポーリングのエッジ検出のみを使用しています。

6. メインループと状態遷移

6.1 入力処理

// ボタン状態のビットマスク化
var cur: UInt8 = 0
if Buttons.left  { cur |= 1 }
if Buttons.right { cur |= 2 }
if Buttons.a     { cur |= 4 }
if Buttons.b     { cur |= 8 }
if Buttons.up    { cur |= 16 }
if Buttons.down  { cur |= 32 }

// エッジ検出: ポーリング + ISR
let polledPressed = cur & ~lastButtons      // ポーリングの立ち上がりエッジ
let isrPressed = ButtonISR.consumePressed()  // ISRがキャプチャしたイベント
let pressed = polledPressed | isrPressed
lastButtons = cur

入力方式の使い分け

操作検出方法理由
回転 (A/B)polledPressed のみISRバウンスで二重回転を防止
ハードドロップ (Up)polledPressed のみ + Down非押下チェックバウンス防止 + ポーズ操作との衝突回避
左右移動 (DAS)cur（保持状態）120ms間隔のリピート入力
ソフトドロップ (Down)cur（保持状態）50ms間隔の高速落下
ポーズ (Left+Right)cur（同時押し判定）保持状態の組み合わせ

各状態の詳細

State 0: タイトル画面

  • TETRISロゴ（フレーム付き）
  • デモプレイAI（簡易的に自動プレイ、3pxセル縮小ボード）
  • 「PRESS START」点滅テキスト
  • BGMは鳴らない

State 1: プレイ中

  • 6pxセルの10×20ボード + ボーダー
  • スコア表示（左上）、レベル表示（右上）、ネクストピース（右上小）
  • BGM再生（SysTick割り込み駆動）
  • フレーム間隔: 16ms（~60fps目標）

State 4: ポーズ

  • 左右ボタン同時押しで発動
  • 盤面を隠す（「PAUSE」を3倍スケール表示）
  • 全ボタン離す → 任意ボタンで復帰
  • 復帰時にDAS/ソフトドロップのタイマーをリセット

State 5: ライン消去アニメーション

  • 80ms間隔で8フェーズ（4回点滅）
  • SysTick割り込みのおかげで、I2Cフラッシュ中も正確なタイミング
  • これがハイブリッドアーキテクチャの真価を発揮する場面

State 3: ゲームオーバー画面

  • 「GAME OVER」3倍スケール文字
  • スコア（2倍スケール、中央揃え）
  • ライン数・レベル表示
  • 「PRESS START」点滅

7. サポート層

7.1 ベクターテーブル

ARM Cortex-M0+ のベクターテーブルを Swift のタプルリテラルで定義します。@section(".vector") でリンカスクリプトの .vector セクションに配置されます。

@used

遭遇した問題：64ビット演算の無限再帰
__aeabi_llsl の実装で UInt64 << 32 を使うと、コンパイラがそれを __aeabi_llsl に展開し無限再帰になりました。makeUInt64(lo:hi:) でポインタ操作を使い、64ビット演算を一切使わずに UInt64 を構築することで解決しています。 
// ポインタ操作で UInt64 を構築（64ビット演算を回避）
private func makeUInt64(lo: UInt32, hi: UInt32) -> UInt64 {
  var result: UInt64 = 0
  withUnsafeMutablePointer(to: &result) { ptr in
    let p = UnsafeMutableRawPointer(ptr).assumingMemoryBound(to: UInt32.self)
    p[0] = lo  // リトルエンディアン: 下位32ビットがアドレス0
    p[1] = hi
  }
  return result
}

9. 開発中に遭遇した問題と解決策

SysTick 変数のキャッシュ問題

症状: ゲームがフリーズ。落下もアニメーションも止まる。

原因: コンパイラが _tick をレジスタにキャッシュし、割り込みによる更新が見えない。

解決: withUnsafePointer + loadUnaligned でメモリから直接読み取り。

Music.playing の揮発性問題

症状: 効果音が鳴らない。

原因: Music.stop() で playing = false にしても、SysTick割り込みが古い値 true を読み続け、PWMレジスタを上書き。

解決: playing の読み書きをポインタベースの揮発性アクセスに変更。

64ビット演算スタブの無限再帰

症状: スタックオーバーフロー → HardFault。

原因: __aeabi_llsl 内で UInt64 << 32 を使うと、コンパイラがそれを __aeabi_llsl 自身に展開。

解決: makeUInt64(lo:hi:) でポインタ操作によりUInt64を構築。64ビット演算を一切使わない。

I2C IC_TAR の書き込みタイミング

症状: SSD1306の表示が間欠的に壊れる。

原因: IC_TAR レジスタを IC_ENABLE=1 の間に書き込んでいた。

解決: disable → 書き込み → enable のサイクル。アドレスキャッシュで冗長サイクルを回避。

ペリフェラルのリセット不足

症状: デバッグ停止→再デプロイを3回繰り返すと動作しなくなる。

原因: I2C/UART/PWM が前回の状態を保持している。

解決: Board.init() で Resets.reset() してからアンリセット。

パッド初期化のデフォルト値依存

症状: C言語版を先に実行した後でないと動作しない。

原因: PadsBank.enablePullUp() がRead-Modify-Writeで、レジスタのデフォルト値に依存。

解決: PadsBank.store(pin, 0x4A) で直接書き込み。

配列リテラルのヒープ確保

症状: ボタン初期化時にヒープ確保が発生（バンプアロケータのメモリ消費）。

原因: let pins = [2, 3, 4, 5, 6, 7] がEmbedded Swiftでヒープ配列を生成。

解決: 個別の configureButton(pin) 呼び出しに変更。

ボタンISRのバウンス

症状: 回転やハードドロップが意図せず2回発動。

原因: GPIO割り込みがボタンのバウンスで複数回発火し、ISRビットマスクに重複記録。

解決: エッジ検出が重要な操作（回転・ハードドロップ）はポーリングのみで判定。ISRはDAS/ソフトドロップの取りこぼし防止に使用。
